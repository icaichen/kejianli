# 可见力产品路线图与数据真实性门槛

> 修订：2026-07-16。本文是实施顺序的唯一依据；任何「已上线」表述都必须同时满足本文件的数据真实性门槛。

## 北极星与三条产品线

可见力的目标不是生成更多内容，而是让品牌在真实 AI 答案面中被可靠地发现、优化并持续提升。

当前优先 ICP 是企业品牌/市场研究团队与服务这些客户的咨询公司。产品的一级工作单位是“客户下的市场研究项目”，至少包含市场、品类、核心品牌、竞品、研究目标和固定问题范围；网站审计属于辅助证据，不是企业用户的默认入口。

新建项目时可以读取品牌官网，辅助提取页面明确声明的品牌名、品类与描述，并向用户展示每一项的网页来源。它不能替代研究 Brief，也不能从自有官网猜测竞品；所有建议必须由用户确认后才进入项目范围。

1. **检测与追踪（Observe）**：审计网站是否具备被理解与引用的条件；观测真实 AI 答案中的品牌、引用和竞品份额。
2. **自助优化（Improve）**：把诊断转化为可审批、可发布、可验证的内容、实体、结构化数据和分发任务。
3. **Agent 托管执行（Operate）**：在用户授权范围内形成“发现差距 → 提议改动 → 审批 → 发布 → 复测”的闭环。

三层必须按此顺序成立。第三层不能用模拟观测数据驱动；第二层的效果也必须回到第一层验证。

## 阶段与放行条件

### Phase 0：产品骨架（已完成）

- 网站审计、评分、建议、项目、工作台、交付工作包与 Provider 抽象。
- Stub 仅用于开发、演示与测试；不得作为真实客户指标或趋势。
- DeepSeek Chat 的直连已验证，但目前只能标为“模型回答采样”，不是搜索引用追踪。

### Phase 1：真实观测数据层（两条答案面已达到首个放行门槛，仍持续加固）

目标：让每一个显示给用户的指标都能回溯到真实回答与采集条件。

必须交付：

- **Provider 资格矩阵**：每个引擎标明 `采集方式 / 产品表面 / 联网状态 / 地区语言 / 登录状态 / 引用可得性 / 最近验收时间 / 成本`。
- **证据仓库**：保存原始回答、引用 URL、查询参数、时间戳、模型/产品版本和采集记录；UI 可查看证据。
- **指标分级**：
  - `品牌认知`：普通聊天 API 的品牌提及；不能叫 citation/搜索排名。
  - `答案面可见度`：来自实际搜索/联网回答面的品牌提及和位置。
  - `引用可见度`：答案面明确返回或展示的自有域名来源链接。
- **真实 Provider 验收**：每个 Provider 用固定品牌/问题集与人工查看的产品界面对照；达到一致性阈值后才允许进入正式报表。
- **抽样科学性**：问题集涵盖品牌、品类、问题、比较意图；设置重复次数、缓存窗口、置信区间与失败重试。
- **研究问题设计**：项目 brief 自动生成四类企业研究问题，默认提供 16 个候选、选择 12 个作为首轮范围，并允许在采样前编辑或取消。
- **持续执行**：追踪计划保存问题集、引擎、样本数和频率；部署调度器定期调用 `python -m keeplix.jobs.tracking`，手动执行与到期执行共用同一数据链。
- **企业研究报告**：按每个答案面的最新正式运行聚合管理层摘要、品牌/竞品表现、来源结构、研究方法与范围限制；支持浏览器打印/PDF 交付。
- **明确标识**：Stub 永远不能与真实样本合并；失败、降级与不可用必须在 UI 中可见。

第一批只选择有合法稳定数据路径的 2–3 个 AI Surface。先达到可信，再扩覆盖；不要把“有模型名称”当作“已支持”。

截至 2026-07-16，千问联网检索 Agent 与百度智能搜索生成已完成真实回答、来源、请求标识、人工验收和正式报告链路，满足 Phase 2 开始条件。Phase 1 并未因此结束：问题覆盖、重复采样、定时执行、更多答案面验收和长期稳定性仍需持续建设，且不得用未验收 Provider 补足表面数量。

当前实现已为每条问题保存意图，并在问题集、采样运行和正式趋势中冻结覆盖质量快照。覆盖不足不会被隐藏：结果会标为 `limited` 或 `balanced` 并列出缺失意图；它仍可作为该问题范围内的真实观测，但不能被解释为全面市场覆盖。

项目 Brief 是问题设计之前的版本化研究资产。品牌、域名、市场、品类、竞品或研究目标变化时，旧问题集和追踪计划只保留为历史证据，不能继续代表当前范围；旧计划自动暂停，用户必须基于新 Brief 确认问题集后再建立基线。

正式研究的 Brief 完整性由后端统一判定：核心品牌、市场、明确品类、至少一个竞品和研究目标缺一不可。门槛未满足时不生成企业问题框架、不保存正式问题集、不建立追踪计划；即使通过低层采样 API 保留开发证据，也不得写入正式趋势或报告。

每次正式采样和可见度快照都冻结当时的 `brief_version`。Brief 更新后，项目概览、当前诊断和企业报告立即回到“等待新基线”；旧回答与来源仍作为历史证据保留，但不能与新研究口径混算。

项目内的“研究报告”视图只读取 `report_eligible` 的最新正式证据，不把 Stub、失败运行或同一答案面的旧批次混入当前交付。没有正式基线时报告明确停留在等待状态，不生成看似完整的结论。

研究报告同时公开当前 `brief_version`、问题数、问题覆盖质量、问题集与追踪计划标识，保证客户看到的结论可以回到同一研究范围，而不是只看到脱离采集口径的百分比。

报告必须区分“总体提及率”和“不含品牌名的自然发现率”。品牌题与比较题会主动提供品牌名，不能用来证明品牌在品类或需求问题中被自然发现；管理层摘要优先展示品类与需求问题的发现率，并同时保留四类意图明细。

正式趋势只允许在同一追踪计划、同一答案面、同一采集方式与同一测量范围内计算。单次检测只作为当前证据；答案面或采集范围改变时必须建立新基线，不能显示升降。项目 API 会为每个正式快照返回 `standalone / baseline / comparable / scope_changed` 判定，前端只对 `comparable` 展示变化值。

建立正式基线时，前端必须先冻结当前 Brief 对应的问题集与追踪计划，再用该 `tracking_plan_id` 执行首次采样；不能先采样后建计划，否则首次结果无法成为后续可比基线。品牌、域名与竞品只能从当前 Brief 读取，执行前必须展示问题数、答案面、重复次数和预计答案样本总量，并由用户明确确认。失败重试复用同一问题集与追踪计划，不得重复制造研究资产。

### Phase 2：自助 GEO 优化工作台（当前开发主线）

- 以审计和真实观测差距生成任务：页面结构、FAQ、事实证据、实体资料、JSON-LD、内容更新与分发。
- 项目级「已核验品牌事实」是 Improve 与 Agent 的唯一事实输入；每条事实必须保存公开来源 URL 和审核状态。
- 证据化生成必须同时绑定当前 Brief 的正式诊断、已核验品牌事实和实际生成引擎；旧 Brief 工作项不得生成当前产物。
- 草稿需保存证据快照、事实 ID、生成引擎、版本、编辑和批准状态；绝不自动制造虚假事实或虚假外部提及。
- 用户修订必须继承上一版本的诊断、事实与生成来源；审批、导出和实施前再次检查 Brief 版本与事实审核状态，来源过期或事实停用时必须阻断。
- 发布/导出渠道、任务负责人、完成状态；本周期工作完成或明确跳过后，最后一次实施自动建立持久化复测计划，并由到期任务复用原测量口径执行。
- 周期归因只有在 Brief、问题、答案面、采集方式、测量范围和正式样本数一致时才显示变化；任一字段漂移都将结果标为不可比并要求建立新基线。
- 归因视图：改动 → 复测 → 指标变化，并明确相关性不等于因果。

### Phase 3：Agent 托管执行（Phase 2 可验证后）

- Agent 只在用户设置的权限、预算、渠道和审批策略内运行。
- 循环：发现 → 计划 → 生成可审计草稿 → 用户审批 → 发布/派发 → 复测 → 报告。
- 需要：任务队列、长任务恢复、版本/回滚、审批日志、成本上限、告警、失败隔离与人工接管。
- 绝不允许 Agent 伪造引用、刷量、绕过平台条款或未经授权发布。

### Phase 4：规模化与商业化（最后）

- 鉴权、多组织、计费、配额、审计日志、团队协作、API、稳定部署与观测。
- 增加更多经过验收的 Provider、行业问题模板、竞品基准与合作/数据供应商。

## 任何新功能的检查问题

在开发前先问：它使用的是哪一类真实证据？结果能否让客户复查？没有 Phase 1 的真实证据，就只能被标记为开发演示，不能成为自动优化或商业决策的依据。
